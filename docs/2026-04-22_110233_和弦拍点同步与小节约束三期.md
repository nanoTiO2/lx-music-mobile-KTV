不包含：

## 3. 本轮架构拆分

### 3.1 保留的既有链路

1. 音频切段得到 `ChordSegment`
2. 拍点分析得到 `BeatAnalysis`
3. 将和弦结果对齐到 beat grid
4. 基于候选分数做一次 DP 平滑

### 3.2 本轮新增的后处理职责

在 beat-grid DP 之后，新增一层“按小节限制变化密度”的后处理：

1. 统计每个小节的和弦变化次数
2. 对变化过密的小节，寻找最弱的短 run
3. 把这个 run 向前或向后邻接和弦吸收
4. 重新做一次单拍抖动清理

这样把职责拆开：

- 拍点同步：负责“变换落在哪一拍”
- DP：负责“候选序列整体最顺”
- 小节约束：负责“不要在一个小节里碎成很多段”

## 4. 代码改动

改动文件：

- `android/app/src/main/java/cn/toside/music/mobile/mixer/MixerModule.java`

### 4.1 拍点同步继续沿用 ChordMiniApp 思路

`synchronizeChordSeriesToBeats(...)` 继续使用 onset 到 beat 的双指针同步逻辑：

- 不再按每拍去反查“重叠最多的段”
- 而是按和弦起点顺序往后推进
- 当起点落在“下一拍前 35% beatDuration”区间时，映射到下一拍
- 再 forward fill

这部分更接近 `ChordMiniApp` 的 `chordSynchronization.ts`

### 4.2 降低强拍结构奖励

`computeBeatPlacementBonus(...)` 中：

- 下拍、强拍上的调内奖励整体降低
- 一级/五级的额外偏置缩小

目的：

- 避免 DP 过度偏向主功能和弦
- 减少把中间真实过渡和弦吞掉，最终又回成“几乎全程一个和弦”

### 4.3 新增小节内变和弦密度约束

新增方法：

- `limitChordChangesWithinBars(...)`
- `countChordChanges(...)`
- `findWeakestRunStartInBar(...)`
- `computeAverageCandidateScore(...)`
- `collapseRunTowardNeighbor(...)`

逻辑：

1. 以拍号为单位扫描小节
2. `4/4` 默认最多保留 2 次变化，`3/4` 默认最多保留 1 次变化
3. 如果超限，挑出当前小节里“最弱的 run”
4. 最弱 run 的判断依据：
   - 当前 run 在候选分数里的平均支持度
   - 与前/后邻接和弦的竞争差
   - run 时值
   - 是否落在下拍/强拍
   - 与邻接和弦的过渡合理性
5. 将该 run 吸收到前一个或后一个邻接和弦

这样做的目标不是“硬编码某几个和弦”，而是让结果更像真实弹唱分配：

- 弱拍短促变化更容易被吸收
- 小节开头和强拍上的变化更容易保留
