# LX Music 二开架构设计

## 当前现状

- 代码仓库位于 `lx-music-mobile-master/lx-music-mobile-master`。
- 当前播放器基于 `react-native-track-player`，歌词同步链路和横竖屏播放页已存在。
- 下载页目前仍是占位，下载入口和真实保存任务未闭环。
- 现有日志仅有文本日志能力，尚未满足 SQLite 过程日志要求。
- 本地媒体与文件系统工具已存在，可作为后续“下载落地”和“本地聚合导入”的复用基础。

## 模块拆分

- 日志与任务编排层
  负责 `codex_lx二开_log.db`、阶段文档、脚本入口、运行记录。
- 下载与保存层
  负责在线资源保存、本地目录记忆、任务状态、歌词同名保存。
- 本地媒体聚合层
  负责扫描目录并把原曲、歌词、伴奏、鼓点等变体聚合为逻辑歌曲。
- 播放与切源层
  负责逻辑歌曲播放状态、变体切换入口、后续 Android 原生高保真切源桥接。
- 歌词显示层
  负责标准歌词、满屏歌词、提词器模式与横竖屏适配。
- 测试与文档层
  负责增量验证、阶段说明、变更留痕。

## 数据流

1. 用户在列表或播放页触发“保存到本地”。
2. 下载层解析保存路径并创建任务。
3. 文件落地后更新本地媒体记录。
4. 本地媒体聚合层将同目录相关文件组织为逻辑歌曲。
5. 播放层按“逻辑歌曲 + 当前变体”驱动播放器与歌词页。
6. 关键节点统一写入 SQLite 日志库和阶段文档。

## 子任务拆分

- 分析 Agent
  继续核对下载、目录选择、播放器桥接复用点。
- 编排 Agent
  落地 SQLite 日志库、`main.py`、`run.bat`、文档与脚本入口。
- 下载 Agent
  接入保存目录配置、保存任务状态和下载列表页骨架。
- 聚合 Agent
  实现本地目录扫描与原曲/歌词/伴奏聚合。
- 播放 Agent
  接入变体切换 UI 与逻辑歌曲播放状态。
- 测试 Agent
  补充脚本测试和关键流程验证。

## 执行顺序

1. 先补日志基础设施与阶段文档。
2. 再落下载保存目录与任务骨架，形成第一个可运行增量。
3. 随后实现本地聚合导入。
4. 再接逻辑歌曲播放与切源入口。
5. 最后推进原生高保真切源和歌词大屏模式。
