## 3. 关键实现

### 3.1 和弦后处理增强

在 `MixerModule.java` 中，本轮把原来的“切段 -> 拍点对齐 -> 直接重建段”升级为：

1. 先根据 beat 区间为每一拍收集多个候选和弦分数：
   - 区间 overlap
   - 段中心与拍中心距离
   - 原段置信度
   - 调内结构分
   - 强拍/下拍位置奖励
2. 再在 beat 级别做一次动态规划，依据：
   - 当前拍候选分
   - 前后和弦转移分
   - 调内功能关系
   - 强拍位置偏好
3. 最后增加一层“孤立 1 拍抖动”抑制，吸收前后同和弦夹着的单拍噪声。

对应新增/调整的核心方法：

- `buildBeatCandidateScores(...)`
- `computeBeatPlacementBonus(...)`
- `decodeChordLabel(...)`
- `computeTransitionBonusByLabel(...)`
- `buildBeatCandidateRows(...)`
- `refineChordSeries(...)`
- `suppressIsolatedBeatFlutters(...)`

`alignChordSegmentsToBeatGrid(...)` 也改成接受 `keyRoot / mode`，使 beat 级修正能复用调性信息。

### 3.2 简谱改为子拍级表达

`KeyInfoBtn.tsx` 中新增：

- `getJianpuSubdivisionCount(...)`
- `buildPitchSlots(...)`
- `PitchSlotInfo`

处理方式：

- `4/4`、`3/4`：
  - 每拍细分为 2 个子拍，形成更像八分音符级别的文本。
- `6/8`：
  - 保持每拍 1 个 token，避免已经是 6 拍时再次过度分裂。

新的简谱 token 规则：

- `1 2 3 4 5 6 7`：当前子拍主音级
- `#1 / b3` 等：升降级
- `-`：延音/持续到下一子拍
- `0`：该子拍未抓到有效音高

最终一个小节会变成类似：

```txt
01|1234 | 1- 23 5- 60 |
   | 你  好  世  界 |
```

这样比“每拍只给一个音”更接近节奏型，同时仍保持纯文本可读。

### 3.3 封面写回状态持久化

在下载任务 `metadata` 中新增：

- `coverEmbedStatus?: 'idle' | 'embedded' | 'failed'`
- `coverEmbedError?: string | null`
- `coverEmbedUpdatedAt?: number`

实现策略：

1. 新建下载任务时默认 `idle`
2. `saveTaskCover()` 或同步导入阶段尝试 `writePic()`
3. 成功则标记 `embedded`
4. 失败则标记 `failed`，记录错误文本与更新时间
5. 后续 `syncDownloadedList()` / `importDownloadedMusicToDownloadList()` 只有在状态不是 `embedded` 且不是 `failed` 时，才继续自动尝试

这样历史任务只会在没有状态时自动补一次，而不是每次扫描都重试。

## 4. 与参考项目的借鉴关系

本轮继续借了 `ChordMiniApp` 的两类思路，但没有照搬其整体服务端架构：

1. 借“拍点同步后再做视觉/结构修正”的思路，而不是把原始切段直接展示。
2. 借“调性与强拍位置要进入和弦修正”的思路，而不是只看单段局部 chroma 分数。

由于当前项目是本地离线 RN Android 方案，而 `ChordMiniApp` 深层依赖服务端模型与更复杂的周边流水线，因此本轮依旧采用“在现有 native 分析链上做可运行的后修正增强”路线。
