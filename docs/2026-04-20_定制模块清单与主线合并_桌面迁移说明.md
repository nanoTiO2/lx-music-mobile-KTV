# 定制模块清单与主线合并 / 桌面迁移说明

时间：2026-04-20 00:00 +08:00

## 1. 你目前这套二开，和原始源码的主要差异

这次二开不是只改了一个点，而是形成了 6 组明显的定制能力：

### A. 低版本 / 魔改 Android 存储权限兼容

- 目标：解决 Android 7 到 Android 13 上本地目录导入与授权不稳定的问题。
- 主要文件：
- `src/core/common.ts`
- `src/utils/tools.ts`
- `src/components/common/ChoosePath/index.tsx`
- `android/app/src/main/AndroidManifest.xml`
- 与原始源码的差异：
- 现在更强调按 Android 版本分别请求权限，并把目录选择和授权入口统一到应用内流程，而不是假设用户一定能从系统设置手动授权。

### B. 本地导入缓存与离线优先

- 目标：本地歌曲导入后缓存元数据，不反复“快速加入再读取标签”；已有本地 LRC / 元数据时默认不联网。
- 主要文件：
- `src/screens/Home/Views/Mylist/MyList/listAction.ts`
- `src/core/music/local.ts`
- `src/core/music/index.ts`
- `src/components/MetadataEditModal/MetadataForm.tsx`
- `src/lang/zh-cn.json`
- `src/lang/zh-tw.json`
- `src/lang/en-us.json`
- 与原始源码的差异：
- 现在本地逻辑是“离线优先、手动联网”，并且会尽量复用已导入文件的元数据缓存，而不是每次重新读标签或默认联网补全。

### C. 原生混音器：变调、变速、KTV 与本地播放稳定性

- 目标：让本地播放支持变调和更稳定的切换/过渡，尤其是安卓设备上的实际可用性。
- 主要文件：
- `android/app/src/main/java/cn/toside/music/mobile/mixer/MixerModule.java`
- `android/app/src/main/java/cn/toside/music/mobile/mixer/MixerPackage.java`
- `src/utils/nativeModules/mixer.ts`
- `src/plugins/player/utils.ts`
- `src/screens/PlayDetail/components/SettingPopup/settings/SettingPitch.tsx`
- `src/screens/PlayDetail/components/SettingPopup/settings/SettingPlaybackRate.tsx`
- 与原始源码的差异：
- 你的版本已经不只是 TrackPlayer 直放，而是加了一层 Android 原生 mixer，负责 pitch / playback rate / 过渡和 seek 稳定恢复。

### D. 手动调号 / 节拍分析与 LRC 写回

- 目标：把高耗能分析改成手动触发，给出进度反馈，并把结果保存到 LRC 头部，方便下次直接读取。
- 主要文件：
- `src/core/musicProfile.ts`
- `src/core/music/local.ts`
- `src/screens/PlayDetail/components/KeyInfoBtn.tsx`
- `android/app/src/main/java/cn/toside/music/mobile/mixer/MixerModule.java`
- 与原始源码的差异：
- 原始逻辑更偏向即时计算；现在你的版本是“手动分析 + 缓存 + LRC 写回 + 下次优先复用”。

### E. 歌词舞台投影模式

- 目标：服务投影仪和手机投屏场景，提供独立舞台页面、纯黑底、菜单自动隐藏、跑马灯配色、手动切屏、全屏沉浸和中文字体。
- 主要文件：
- `src/screens/LyricStage/index.tsx`
- `src/screens/PlayDetail/components/LyricStageBtn.tsx`
- `src/navigation/screenNames.ts`
- `src/navigation/navigation.ts`
- `src/navigation/registerScreens.tsx`
- `src/utils/nativeModules/utils.ts`
- `android/app/src/main/java/cn/toside/music/mobile/utils/UtilsModule.java`
- `android/app/src/main/assets/fonts/NotoSansCJKsc-Regular.otf`
- 与原始源码的差异：
- 这已经是一个明显的新增产品能力，而不是简单调样式。

### F. 播放中拖动进度条与歌词同步修复

- 目标：解决底部进度条拖动卡顿、位置回跳、歌词不同步。
- 主要文件：
- `src/store/player/action.ts`
- `src/core/init/player/playProgress.ts`
- `src/core/init/player/lyric.ts`

- `src/components/player/ProgressBar.tsx`

- 与原始源码的差异：
- 你的版本加入了 seek 版本控制、seek 后位置回读、歌词强制重同步，以及原生 mixer 的 seek 状态收口。

## 2. 已形成的独立模块

为了避免以后忘记哪些地方是二开定制，我已经新增一个独立模块清单文件：

- `src/customModules/moduleManifest.ts`

这个文件不是运行时依赖，而是你自己的“定制模块总表”。里面已经把每个模块的：

- 模块 ID
- 功能说明
- 涉及平台
- 涉及源码文件
- 主线合并风险
- 桌面迁移提示

都固定下来了。

以后你再做二开，优先把新功能也补到这个清单里，而不是只散落在聊天记录或日志数据库里。

## 3. 以后主开源分支更新，你应该怎么合并

当前目录没有 `.git`，所以不能直接做标准 upstream merge。以后建议改成下面这套流程：

### 第一步：保留一份“纯净上游源码”

- 新建两个目录：
- `lx-mobile-upstream/`：只放原始开源仓库，不做你的定制改动
- `lx-mobile-custom/`：你的二开版本

### 第二步：把你的定制改动按模块管理

- 所有定制点都先登记到 `src/customModules/moduleManifest.ts`
- 每次较大改动都补一份 `docs/时间戳_说明.md`
- 每轮功能完成后写入 SQLite 日志数据库

### 第三步：主线更新时不要整仓硬覆盖

推荐顺序：

1. 先更新 `lx-mobile-upstream/` 到新的开源版本
2. 对照 `moduleManifest.ts`，按模块检查影响文件
3. 优先合并“低风险模块”
4. 再处理“高风险模块”

建议按这个顺序：

1. 本地导入缓存与离线优先
2. 权限兼容
3. 手动分析与 LRC 写回
4. 歌词舞台投影
5. seek 同步
6. mixer 原生层

原因：

- 前 3 类主要是业务逻辑修改，冲突相对可控
- 后 3 类更依赖播放器、导航、原生桥，最容易和主线发生结构冲突

### 第四步：碰到播放器主线大改时，优先保接口，不保实现

尤其是下面几个文件，如果主线改动很大，不要硬抄旧实现：

- `src/plugins/player/utils.ts`
- `src/core/init/player/playProgress.ts`
- `src/core/init/player/lyric.ts`
- `android/app/src/main/java/cn/toside/music/mobile/mixer/MixerModule.java`

正确做法是：

- 先保留你自己的接口契约
- 再用新主线的结构重新接进去

你当前应该保的接口契约包括：

- 本地播放默认不联网补歌词/封面
- 手动分析才跑重计算
- 分析结果可写回 LRC
- 变调和变速要继续工作
- 歌词舞台要保留纯黑投影、字体、横竖屏和低耗模式
- 底栏拖动后音频与歌词必须能重新同步

## 4. 如果要把这些功能搬到电脑端，应该怎么做

不建议一次性整包照搬。应该按模块拆。

### 适合优先搬到电脑端的模块

#### 1. 本地导入缓存与离线优先

- 这是最容易迁移的
- 主要是 TypeScript 业务逻辑
- 基本不依赖 Android

#### 2. 手动调号 / 节拍分析与 LRC 写回

- 协议层可以直接复用
- LRC 头部标签 `lx_music_profile` 也可以继续沿用
- 只需要把原生分析执行层替换成桌面端实现

#### 3. 歌词舞台投影模式

- UI 与交互最值得迁移到电脑端
- 电脑端大屏展示反而更合适
- 页面状态管理和大部分 React 渲染逻辑可以复用

### 不适合直接搬的模块

#### 1. Android 原生 mixer

- `MixerModule.java` 不能直接给电脑端用
- 桌面端必须换成新的底层音频实现
- 如果电脑端是 Electron，通常要用：
- Node 原生音频库
- FFmpeg / mpv / VLC / libsoundtouch 一类底层能力

#### 2. Android 权限与状态栏沉浸

- 这类功能要全部替换成桌面平台 API

## 5. 电脑端迁移建议架构

如果未来要把这些功能加到电脑端，建议按下面拆：

### shared-core

- 本地导入缓存
- 本地歌词读取
- 元数据回填
- 分析结果缓存
- LRC 写回协议

### shared-ui

- 歌词舞台页面状态
- 颜色主题
- 字体选择
- 菜单显示/隐藏
- 跑马灯配色逻辑

### desktop-adapter

- 文件系统访问
- 桌面窗口全屏
- 屏幕方向替代方案
- 原生音频引擎适配
- 进度条 seek 回调适配

### mobile-adapter

- Android 权限
- Android 原生 mixer
- 状态栏/沉浸模式
- 设备震动与低耗策略

## 6. 实操建议：以后如何维护

以后不要再把“二开能力”散在几十个文件里却没有索引。建议固定执行：

1. 新增功能时，先在 `src/customModules/moduleManifest.ts` 登记模块
2. 大版本完成后，补一份 `docs/时间戳_说明.md`
3. 构建交付前，更新 SQLite 日志
4. 如果主线更新，先看 `moduleManifest.ts` 再动手合并

这样以后无论是你自己继续改，还是交给别人接手，都会清楚：

- 哪些是上游原始功能
- 哪些是你自己的定制能力
- 哪些能迁移到桌面
- 哪些必须按平台重写
